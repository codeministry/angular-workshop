🔍 Das Problem – Warum Signals?

❌ Bisher: Zone.js

Zone.js patcht alle async-APIs global (setTimeout, Promise, fetch…)

Bei jeder Änderung: Angular prüft den gesamten Komponentenbaum

Probleme:

  • Unvorhersehbares Rendering
  • Performance-Overhead bei großen Apps
  • Schwer zu debuggen
✅ Heute: Signals

Signals = feingranularer Reaktivitätsgraph

Angular weiß genau welche Komponente welches Signal liest

Vorteile:

  • Nur betroffene Komponenten werden neu gerendert
  • Deterministisch und debuggbar
  • Kein Zone.js mehr nötig (ab Angular 18+)
Spring-Analogie: Zone.js = @Scheduled-Task der den gesamten ApplicationContext bei jedem Tick neu auswertet. Signals = gezielter @EventListener der nur auf relevante Events reagiert.

import { signal } from '@angular/core';

@Component({ /* ... */ })
export class KanbanBoardComponent {

  // Signal erzeugen – wie ein "smarter" Wert
  showForm = signal(false);       // Initialwert: false
  taskCount = signal(0);

  toggle() {
    // Wert lesen: Signal aufrufen wie eine Funktion
    console.log(this.showForm());   // false

    // Wert setzen
    this.showForm.set(true);

    // Wert transformieren
    this.showForm.update(v => !v);
  }
}

🔢 computed() – Abgeleiteter Zustand

import { signal, computed } from '@angular/core';

export class KanbanBoardComponent {

  taskCount = signal({ todo: 5, inProgress: 2, done: 3, total: 10 });

  // computed() = abgeleiteter Wert, wird automatisch aktualisiert
  // Spring-Analogie: @Transient berechnetes Feld in einer Entity
  completionPercent = computed(() => {
    const { done, total } = this.taskCount();
    return total > 0 ? Math.round((done / total) * 100) : 0;
  });

  // Nur neu berechnet wenn taskCount() sich ändert!
  // Gecacht wie Memoization in Java (Guava Cache, etc.)
}
Template: {{ completionPercent() }}% – kein zusätzlicher Code!

⚖️ Signals vs. Observables – Wann was?

Kriterium Signal Observable
UI-Zustand (Formular offen?) ✅ Ideal Overkill
HTTP-Request Nur via toSignal() ✅ Ideal
Abgeleiteter Wert (Berechnung) ✅ computed() map()-Operator
WebSocket / Event-Stream Nur via toSignal() ✅ Ideal
NGXS Store-Selektor ✅ via toSignal() ✅ direkt möglich
Debounce / Throttle Umweg nötig ✅ debounceTime()
toSignal(obs$) → Observable → Signal
toObservable(sig) → Signal → Observable
Faustregel: UI-State → Signal. Async-Datenströme → Observable. Brücke bei Bedarf: toSignal().

🎯 @Directive – PriorityBorderDirective

@Directive({
  selector: '[appPriorityBorder]',   // CSS-Attribut-Selector
  standalone: true,
})
export class PriorityBorderDirective implements OnChanges {

  // input() mit Alias – Angular 21 Modern Pattern (kein @Input() mehr)
  readonly priority = input<TaskPriority>('low', { alias: 'appPriorityBorder' });

  private readonly el = inject(ElementRef);
  private readonly renderer = inject(Renderer2);

  private readonly colorMap = {
    high: '#dc3545', medium: '#fd7e14', low: '#198754'
  };

  ngOnChanges(): void {
    // Signal lesen mit () – Renderer2 für SSR-sichere DOM-Manipulation
    this.renderer.setStyle(
      this.el.nativeElement, 'border-left',
      `4px solid ${this.colorMap[this.priority()]}`
    );
  }
}
Verwendung: <div [appPriorityBorder]="task().priority">
→ task() weil task ein Signal-Input ist!
Beachte: priority() statt priority
→ Signal-Input = Funktion!

🔍 @Selector – Gecachte Abfragen

export class TaskSelectors {

  // Einfacher Selektor – liest direkt aus dem State
  @Selector([TaskState])
  static allTasks(state: TaskStateModel): Task[] {
    return state.tasks;
  }

  // Berechneter Selektor – wie SQL WHERE-Klausel
  @Selector([TaskState])
  static filteredTasks(state: TaskStateModel): Task[] {
    const q = state.filter.toLowerCase();
    return state.tasks.filter(t =>
      (!q || t.title.toLowerCase().includes(q))
    );
  }

  // Kompositions-Selektor – basiert auf filteredTasks
  @Selector([TaskSelectors.filteredTasks])
  static todoTasks(filteredTasks: Task[]): Task[] {
    return filteredTasks.filter(t => t.status === 'todo');
  }
}

// Verwendung in einer Komponente:
todoTasks = toSignal(this.store.select(TaskSelectors.todoTasks));

🔒 HTTP Interceptor – Funktionaler Stil

// loading.service.ts – Signal-basierter Service
@Injectable({ providedIn: 'root' })
export class LoadingService {
  readonly isLoading = signal(false);
  private count = 0;

  start() { this.isLoading.set(!!++this.count); }
  stop()  { this.isLoading.set(!!Math.max(0, --this.count)); }
}

// loading.interceptor.ts – Funktionaler Interceptor (Angular 15+)
// Spring-Analogie: HandlerInterceptor.preHandle() / afterCompletion()
export const loadingInterceptor: HttpInterceptorFn = (req, next) => {
  const loadingSvc = inject(LoadingService); // inject() im Interceptor!

  loadingSvc.start();
  return next(req).pipe(
    finalize(() => loadingSvc.stop()) // Immer ausgeführt (try/finally)
  );
};

🌊 RxJS – Observable Pipeline

// Marble Diagram – Zeitachse →
Quelle:      ----A--------B--------C---------D---->
map(x => x*2):  ----2A-------2B-------2C--------2D---->
filter(x>3):   ---------2B-------2C--------2D---->
take(2):       ---------2B-------2C|  (|= complete)
Wichtige Operatoren
  • map() – Wert transformieren
  • filter() – Werte herausfiltern
  • catchError() – Fehler behandeln
  • retry({ count: 2 }) – Wiederholungsversuch
  • tap() – Seiteneffekt ohne Transformation
Für UI-Eingaben
  • debounceTime(300) – Warten nach letzter Eingabe
  • distinctUntilChanged() – Duplikate überspringen
  • switchMap() – Vorherigen Request abbrechen
  • takeUntilDestroyed() – Auto-Cleanup
  • finalize() – Immer am Ende ausführen

export class TaskFilterComponent {
  private store = inject(Store);

  // Reaktives Formularfeld ohne FormGroup nötig
  searchControl = new FormControl('');

  constructor() {
    this.searchControl.valueChanges
      .pipe(
        debounceTime(300),        // 300ms nach letzter Eingabe warten
        distinctUntilChanged(),   // Nur bei echter Änderung
        takeUntilDestroyed()      // Kein ngOnDestroy nötig! (Angular 16+)

      .subscribe(value => {

  }
}

// tasks.resolver.ts
// Spring-Analogie: @ModelAttribute im Controller – Daten werden
// geladen BEVOR die Handler-Methode aufgerufen wird
export const tasksResolver: ResolveFn<void> = () => {
  const store = inject(Store);

  // selectSnapshot = synchron – kein Subscribe nötig
  const existing = store.selectSnapshot(TaskSelectors.allTasks);

  if (existing.length === 0) {
    return store.dispatch(new LoadTasksFromApi());
  }
  return of(undefined);  // Daten schon vorhanden
};

// app.routes.ts – Resolver registrieren
{

  resolve: { tasks: tasksResolver },  // ← hier einbinden

}

  // Typsicheres FormGroup erstellen
  taskForm = this.fb.group({
    title:       ['', [Validators.required, Validators.minLength(3)]],
    description: [''],
    priority:    ['medium' as TaskPriority, Validators.required],
    assignee:    [''],
  });

  onSubmit(): void {
    if (this.taskForm.valid) {
      this.store.dispatch(new AddTask(this.taskForm.value as ...));
      this.taskForm.reset({ priority: 'medium' });
    }
  }
}

onTaskDropped(event: CdkDragDrop<Task[]>): void {
  if (event.previousContainer !== event.container) {
    const task = event.previousContainer.data[event.previousIndex];
    const newStatus = event.container.id as TaskStatus;
    this.store.dispatch(new UpdateTaskStatus(task.id, newStatus));
  }

🔄 Angular Migrations – Automatische Code-Modernisierung

Was sind Angular Migrations?
  • Angular-CLI-Schematics die bestehenden Code automatisch modernisieren
  • Ein Befehl migriert das gesamte Projekt in Sekunden
  • Vollständig reversibel (Git-History nutzen)
  • Offiziell von Google gepflegt
Spring-Analogie: OpenRewrite – automatische Code-Rezepte für Java (z.B. JUnit 4 → 5, Spring Boot 2 → 3). Gleicher Ansatz, gleicher Nutzen.
Warum das wichtig ist
# Alle Signal-Inputs auf einmal migrieren
# Vorher: @Input() title: string = '';
# Nachher: readonly title = input('');

ng generate @angular/core:signal-inputs

# Alle @Output() zu output() konvertieren
ng generate @angular/core:outputs

# Alle @ViewChild zu viewChild() signal-basiert
ng generate @angular/core:signal-queries

# NgModules → Standalone Components
ng generate @angular/core:standalone
